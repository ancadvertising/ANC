# ANC ERP - القرارات المعتمدة للنسخة الجديدة

آخر تحديث: 2026-07-22

## معمارية النشر

- GitHub repository: `ancadvertising/ANC`
- Cloudflare Pages: استضافة الواجهة والـPWA وPreview Deployments.
- Google Apps Script Web App: API ومنطق الأعمال فقط.
- Google Sheets: قاعدة البيانات.
- Google Drive: المستندات والمرفقات والفواتير والتسليمات.
- لا يتم عرض صفحات النظام عبر `HtmlService` أو روابط `googleusercontent` في النسخة الجديدة.
- لا يتم تعديل Production قبل نجاح Staging واختبارات القبول.

## البيئات

### Staging

- فرع GitHub مستقل.
- Cloudflare Pages Project أو Preview منفصل.
- Apps Script Deployment تجريبي.
- Google Spreadsheet تجريبية.
- Google Drive folder تجريبي.

### Production

- الفرع `main`.
- Cloudflare Pages production deployment.
- Apps Script Deployment ثابت للإنتاج.
- Spreadsheet وDrive للإنتاج.

## الهوية وتجربة الاستخدام

- الألوان الأساسية: الأبيض والأسود والرمادي.
- ألوان النجاح والتحذير والخطر تستخدم للدلالة الوظيفية فقط.
- الواجهة عربية RTL افتراضيًا، مع قابلية إضافة الإنجليزية LTR.
- Mobile First، قابلة للتثبيت كتطبيق PWA.
- قائمة جانبية على الكمبيوتر، وتنقل سفلي مع صفحة "المزيد" على الهاتف.
- لا توجد شاشات بيضاء: كل Route معروف يعرض محتوى أو حالة خطأ مفهومة.
- Skeleton/Loading وOffline Screen وتحديث Service Worker آمن.
- احترام `prefers-reduced-motion`.

## الهوية والدخول

- الدخول بحساب Google وليس باسم مستخدم وكلمة مرور.
- الواجهة المستضافة على Cloudflare تحصل على Google ID Token.
- الخادم يتحقق من صحة الرمز ثم يطابق البريد مع جدول `Users`.
- لا يصل إلى النظام بريد غير موجود أو `Active != TRUE`.
- لا توجد أي حالة افتراضية تمنح دور ADMIN عند فشل التعرف على المستخدم.
- الصلاحيات تُفحص في الباك إند، ولا يعتمد الأمان على إخفاء عناصر الواجهة.

## الهيكل التشغيلي

```text
Client
└── Projects (one or more)
    └── Orders (one or more)
        ├── Paid Ads
        ├── Photography
        ├── Videography
        ├── Editing
        ├── Design
        └── Other Services
```

- كل خدمة هي Order مستقل.
- كل Order يحمل `Client ID` و`Project ID`.
- المشروع لا يخلط بين بيانات أو طلبات عميلين.

## الفوترة وكشف الحساب

- يمكن إصدار فاتورة لمشروع واحد فقط من مشروعات العميل.
- لا يمكن خلط طلبات مشروعين في فاتورة واحدة.
- يمكن أن تحتوي الفاتورة على طلب واحد أو عدة طلبات غير مفوترة من المشروع نفسه.
- ربط الحركة الموجودة بالفاتورة لا ينشئ استحقاقًا مكررًا.
- التغيير المالي بعد الترحيل يُسجل كـAdjustment، ولا يُجرى overwrite صامت.
- الإلغاء أو الاسترداد ينشئ حركة عكسية أو Credit Note؛ لا تُحذف الحركات التاريخية.
- الدفعات مرتبطة بالعميل والمشروع والفاتورة.
- كشف الحساب متاح على مستوى العميل، مع تصفية حسب المشروع أو الطلب.

## الإعلانات الممولة

- `Base Spend = Days × Daily Rate`
- `Internal Cost = Base Spend × Cost Rate`
- `Commission = Base Spend × Commission Rate`
- `Client Sale Price = Base Spend + Internal Cost + Commission`
- `Bank Debit = Base Spend + Internal Cost`
- `Profit = Client Sale Price - Bank Debit`
- الإعدادات الافتراضية: Cost Rate = 1.19 وCommission Rate = 0.2997.
- الحد الأدنى الافتراضي للربح = 300 EGP، مع حد أدنى اختياري للهامش.
- يمكن تجاوز الإعداد لكل إعلان مع تسجيل السبب والمستخدم والتوقيت.
- الإعلان مرتبط بالعميل والمشروع والطلب ومتطلبات العميل.
- الحركات البنكية والإيداعات والتسويات لا تُحذف.

## خصوصية الموظفين والعملاء

- العميل يرى مشروعاته وطلباته وفواتيره ومدفوعاته وكشف حسابه فقط.
- الموظف يرى العمل المسند إليه فقط.
- Media Buyer يرى الميزانية اليومية ومتطلبات التنفيذ ولا يرى التكلفة أو العمولة أو الربح أو رصيد البنك.
- موظف Studio يرى تفاصيل الإنتاج والتقدم والتسليمات دون البيانات المالية.
- الإدارة والمالية تعرض لهما المعلومات وفق صلاحيات RBAC.

## الحوكمة واعتماد التعديلات

- المدير الأساسي ينفذ إنشاء وتعديل وتغيير حالة وأرشفة واستعادة السجلات مباشرة.
- المدير المساعد يرى أدوات الإدارة نفسها، لكن كل تغيير يقدمه يُحفظ كطلب PENDING.
- لا يغير طلب المدير المساعد السجل الأصلي قبل اعتماد المدير الأساسي.
- الاعتماد يطبق التغيير ويسجل المراجع والتوقيت، والرفض يحفظ الطلب دون تنفيذ.
- الأرشفة هي البديل الافتراضي للحذف وتحافظ على التاريخ والروابط؛ الاستعادة ممكنة.
- لا يجوز أرشفة عميل لديه مشروعات مفتوحة قبل إغلاقها أو أرشفتها.
- تحديث الموظف للتقدم والملاحظات وروابط التسليم يُعد تحديثًا تنفيذيًا ولا يكشف أو يغير البيانات المالية.
- محاكاة الأدوار الحالية في الواجهة للتجربة فقط؛ الحماية النهائية يجب أن تُفرض في API.

## تصميم الفاتورة

- قالب A4 مستمد من نموذج ANC المقدم.
- شعار وعلامة مائية وبيانات الشركة.
- رقم الفاتورة وتواريخ الإصدار والاستحقاق والعميل والمشروع.
- بنود الطلبات والكمية والسعر والخصم والضريبة والإجماليات والمدفوع والمتبقي.
- رابط أو QR للدفع.
- إنشاء PDF وحفظه على Google Drive وربطه بالسجل.
